# Playwright 1.61.0 ESM Loader 崩溃问题

在整理自动化脚本的时候类型报了这个错误

```ts
import { LoginPage } from '../../pages/LoginPage';
// TypeError: context.conditions?.includes is not a function
```

### 排查过程

1. **排除业务代码** — `LoginPage` 类、`config` 工具类本身没有调用 `.includes()`，报错与业务代码无关。

2. **定位报错位置** — 在 node_modules 中搜索 `conditions?.includes`：

   ```bash
   grep -r "conditions" node_modules/playwright/lib/ | grep "include"
   ```

   命中两处：

   ```
   node_modules/playwright/lib/common/index.js
   node_modules/playwright/lib/transform/esmLoader.js
   ```

3. **阅读源码** — Playwright 1.61.0 的 ESM 模块解析钩子中：

   ```js
   function resolve(specifier, context, nextResolve) {
     if (context.parentURL && context.parentURL.startsWith("file://")) {
       const filename = import_url.default.fileURLToPath(context.parentURL);
       const resolved = resolveHook(filename, specifier);
       if (resolved !== void 0) {
         specifier = context.conditions?.includes("import")  // ← 炸在这里
           ? import_url.default.pathToFileURL(resolved).toString()
           : resolved;
       }
     }
     const result2 = nextResolve(specifier, context);
     ...
   }
   ```

4. **分析调用链** — `registerESMLoader()` 中：

   ```js
   if (nodeModule.registerHooks && !process.env.PLAYWRIGHT_FORCE_ASYNC_LOADER) {
     nodeModule.registerHooks({ resolve, load }); // ← 走这条路径
     return;
   }
   ```

   Node.js v23.11.0 支持 `module.registerHooks()`，Playwright 走的是这条同步 hooks 路径。

### 根因分析

`registerHooks` 的 `resolve` 钩子中，Node.js 规范定义 `context.conditions` 为 `string[]` 类型（如 `['import', 'node']`）。但在 **Node.js v23.11.0 + Playwright 1.61.0** 组合下，`context.conditions` 实际传入的值不是数组，导致 `.includes()` 调用失败。

**本质原因**：Playwright 1.61.0 刚发布，Node v23 是 Current 实验线，两个"最新"版本撞在一起暴露了兼容性问题。

### 最终处理

- Playwright 降级至 **1.60.0**
- Node.js 切换至 **v22 LTS**
- 验证冒烟测试通过

### 经验总结

1. **版本选型：Node 用 LTS，框架不追最新** — 奇数版本号（v23）是实验线，生产项目选 LTS（v22）
2. **报错先查 node_modules** — 遇到莫名其妙的 TypeError，`grep` 搜 node_modules 快速定位是否第三方 bug
3. **package.json 锁版本** — `^1.61.0` 可能自动升到问题版本，考虑用 `~1.60.0` 或精确版本号
